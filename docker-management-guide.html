The Docker Lifecycle: Why Things Get Left Behind

When you followed the decommissioning steps for a project, you correctly removed the running application and its configuration. However, you noticed that the Docker "installer" file‚Äîthe **Image**‚Äîwas still on your server. This is by design and is key to understanding Docker's efficiency.

Think of it like this:

  • The Image: This is the master blueprint or "installer file" (e.g., `setup.exe`). Docker downloads this once and keeps it.
  • The Container: This is the actual running program, created *from* the Image. The `docker compose down` command uninstalls this program.
  • The Project Folder: This holds your custom settings (`docker-compose.yml`, `config/`). The `rm -rf` command deletes these personal files.

Docker keeps the Image so that if you want to launch the service again, it doesn't have to re-download it from the internet. This is fast and efficient, but over time it can lead to unused Images taking up disk space.

What Are <none> (Dangling) Images?

When you list your images with docker images, you often see entries where the repository and tag are both <none>. These are called "dangling images."

These are most commonly created when you rebuild an image for an application you are developing, like your Flask app. For example:

  1. You build your Flask app. The resulting image is tagged flask_app_1:latest.
  2. You change your Python code and run docker compose up --build -d again.
  3. Docker builds a brand **new** image and gives it the tag flask_app_1:latest.
  4. The **old** image, which is no longer the "latest", loses its tag. It is now a dangling image, shown as <none>.

These images are generally safe to remove as they are no longer associated with any tagged image and are not being used by any containers.

Part 1: The Safe & Easy Cleanup (Pruning)

Docker provides a simple, safe command to clean up most unused resources, including all dangling <none> images.

Cleaning Up Dangling Images

This is the most common and safest cleanup command you will use.

# This command specifically targets and removes all dangling images.
docker image prune

Docker will ask for confirmation, as this action is irreversible. After you confirm with `y`, it will list all the deleted images and tell you how much disk space was reclaimed.

The "Nuke" Option (Use with Caution)

‚ö†Ô∏è Warning: The following command is very powerful. It will remove all dangling images, **AND** all unused images (images not used by any container), **AND** all stopped containers, **AND** all unused networks.

Only use this if you want a major cleanup. It might remove an image you wanted to keep for a stopped project.

# This command cleans up the entire Docker system. The -a flag means "all".
docker system prune -a

Part 2: Removing Specific, Unused Images

Sometimes you want to remove a specific image that is no longer in use but isn't dangling (for example, the old `filebrowser:latest` image).

Step 1: List All Images

First, get a list of all images on your server to find the one you want to delete.

docker images

You will see a list with columns for `REPOSITORY`, `TAG`, and `IMAGE ID`.

Step 2A: Removing by Name and Tag

This is the most common method. You use the full name from the list.

# Example: Removing the old, non-s6 filebrowser image
docker rmi filebrowser/filebrowser:latest

# Example: Removing the old Homer dashboard image
docker rmi b4bz/homer:latest

Step 2B: Removing by Image ID

This is useful for deleting a specific <none> image without pruning all of them, or if an image name is long and complex.

# Example: From your list, one of the dangling images has ID 5eae50f5830e
# You only need the first few unique characters.
docker rmi 5eae

What if an image is in use? If you try to remove an image that is currently being used by a container (even a stopped one), Docker will protect you and show an error message. You must stop and remove the container (`docker compose down`) before you can remove its image.

Part 3: Other Useful Docker Commands

These commands operate at the top level of Docker, outside of any specific `docker-compose` project, and are useful for getting a high-level overview of your system.

# List all running containers across all projects
docker ps

# List ALL containers, including stopped ones
docker ps -a

# Show a live stream of resource usage (CPU, RAM) for all running containers
docker stats

# List all Docker volumes (where persistent data is stored)
docker volume ls

# List all Docker networks (like our shared 'nginx-proxy' network)
docker network ls
